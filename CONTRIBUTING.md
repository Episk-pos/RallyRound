# Contributing to RallyRound

Thank you for your interest in contributing to RallyRound! This document provides guidelines and instructions for contributing.

## Getting Started

1. Fork the repository
2. Clone your fork: `git clone https://github.com/your-username/RallyRound.git`
3. Create a feature branch: `git checkout -b feature/your-feature-name`
4. Make your changes
5. Test thoroughly
6. Commit with clear messages
7. Push to your fork
8. Submit a pull request

## Development Setup

Follow the setup instructions in README.md to get your development environment running.

## Code Style

- Use meaningful variable and function names
- Add comments for complex logic
- Follow existing code formatting
- Keep functions focused and concise

## Areas for Contribution

### High Priority

- **Smart Scheduling Algorithm**: Analyze multiple participants' availability and propose optimal time slots
- **Notification System**: Email or push notifications for topic updates
- **Time Slot Voting**: Allow participants to vote on proposed times
- **Mobile Responsiveness**: Improve mobile experience

### Medium Priority

- **Search and Filtering**: Add search functionality and category filters
- **User Profiles**: Enhanced user profiles with presentation history
- **Discussion Forums**: Pre and post-session discussions
- **Analytics**: Track topic popularity and participation trends

### Low Priority

- **Themes**: Dark mode and custom themes
- **Internationalization**: Multi-language support
- **Export Features**: Export schedules and reports
- **Integration**: Slack, Discord, or other platform integrations

## Testing

Before submitting a pull request:

1. Test all user flows (create topic, express interest, schedule)
2. Verify GunDB synchronization across multiple browser tabs
3. Test Google OAuth and Calendar integration
4. Check responsive design on different screen sizes
5. Verify no console errors

## Pull Request Process

1. Update README.md if you've added features or changed setup
2. Add comments explaining your changes
3. Ensure your code follows the existing style
4. Test your changes thoroughly
5. Write a clear PR description explaining what and why

## Bug Reports

When reporting bugs, please include:

- Clear description of the issue
- Steps to reproduce
- Expected behavior
- Actual behavior
- Screenshots if applicable
- Browser and OS information
- Console errors if any

## Feature Requests

For feature requests, please:

- Check existing issues first
- Provide clear use case
- Explain expected behavior
- Consider implementation complexity
- Discuss backwards compatibility

## Questions?

Feel free to open an issue for any questions or clarifications.

Thank you for contributing to RallyRound!
